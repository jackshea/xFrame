## Context

当前 Agent Bridage 的连接参数配置能力不足，无法稳定支持多环境（本地、测试、预发）之间的快速切换；并且缺少统一日志输出，导致连接失败、重连异常、消息收发问题的定位成本较高。该变更涉及配置管理、运行时连接流程和日志观测三个方面，属于跨模块的运行时增强。

## Goals / Non-Goals

**Goals:**
- 为 Agent Bridage 提供可设置的 IP 与端口配置能力，并在运行时生效。
- 将配置持久化到本地存储，在应用重启后自动恢复。
- 在连接生命周期与消息通道关键路径提供结构化日志，支持问题排查与回归验证。
- 保持现有调用方接口尽量兼容，避免对上层业务造成大范围改造。

**Non-Goals:**
- 不引入新的远端配置下发系统。
- 不改造 Agent 协议本身（消息结构、序列化协议）。
- 不新增复杂日志后端（如远程日志上传），仅复用现有日志基础设施。

## Decisions

1. 配置模型与持久化
- 采用独立配置对象（如 `AgentBridgeEndpointConfig`）封装 `Ip` 与 `Port`，并提供基本合法性校验。
- 使用项目现有可持久化机制（优先复用框架已有配置存储能力）进行读写，避免引入新依赖。
- 启动流程中按“加载持久化配置 -> 校验 -> 回退默认值”的顺序初始化，确保异常配置不会阻断系统启动。

2. 运行时接入方式
- 在 Agent Bridage 初始化阶段注入配置提供者，避免在业务代码中直接访问持久化存储。
- 提供显式的设置接口（如 `SetEndpoint(ip, port)`），在更新成功后立即持久化，并在需要时触发重连。
- 对连接中的配置切换采用“先断开再重连”的一致流程，降低状态错乱风险。

3. 日志策略
- 统一使用框架日志接口（`IXLogger` / `IXLogManager`），避免散落 `Debug.Log`。
- 将日志分级：信息日志（连接建立/关闭、重连开始与成功）、警告日志（配置非法回退、重试中断）、错误日志（连接失败、消息处理异常）。
- 日志上下文包含最少必要字段：目标 `ip:port`、连接状态、异常摘要、消息方向（发送/接收）。

4. 备选方案与取舍
- 备选方案 A：仅在 UI 层保存输入值，不进入运行时配置对象。问题是与连接逻辑耦合高、难测，放弃。
- 备选方案 B：每次启动都使用硬编码默认地址，不做持久化。无法满足用户目标，放弃。
- 最终方案在实现复杂度与可维护性之间平衡，且符合框架现有 DI/日志模式。

## Risks / Trade-offs

- [风险] 持久化中保存了无效地址导致连接异常循环 -> [缓解] 加载时做严格校验，失败回退默认值并输出告警日志。
- [风险] 运行中切换地址触发频繁重连影响稳定性 -> [缓解] 仅在值实际变化时重连，并增加最小重连间隔保护。
- [风险] 日志过多影响性能和可读性 -> [缓解] 控制日志级别与关键路径采样，避免在高频循环中输出冗余信息。

## Migration Plan

1. 新增配置模型与默认值策略，不改变既有调用入口。
2. 接入持久化读写，并在初始化路径启用回退机制。
3. 接入日志埋点，先覆盖连接生命周期，再覆盖消息收发异常路径。
4. 补充 EditMode 测试并进行回归验证。

回滚策略：若出现兼容性问题，可临时关闭新配置读取逻辑并回退到默认地址，同时保留日志增强。
