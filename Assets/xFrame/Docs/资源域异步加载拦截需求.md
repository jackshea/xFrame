# 资源域异步加载拦截需求

## 1. 文档目的

本文档用于整理 xFrame 当前资源管理体系下，对“资源域销毁后 Addressables 异步加载结果拦截与回收”能力的需求定义。

本文档只覆盖 C# 框架层，不处理 Lua 回调、Lua 引用生命周期和跨语言桥接问题。

## 2. 项目现状

结合当前仓库实现，资源管理相关现状如下：

- 当前统一入口为 [IAssetManager](/mnt/c/UnityProject/xFrame/Assets/xFrame/Runtime/ResourceManager/IAssetManager.cs)，仅提供全局 `LoadAssetAsync` / `ReleaseAsset` 能力。
- 当前默认实现为 [AddressableAssetManager](/mnt/c/UnityProject/xFrame/Assets/xFrame/Runtime/ResourceManager/AddressableAssetManager.cs)，通过 `_inflightLoads` 以 `address` 维度合并并发请求。
- 当前缓存与释放模型是“全局单例 + address 映射 + LRU 缓存”，没有资源域、请求代际、请求级 token、完成回调拦截闸门。
- 当前异步加载完成后会直接进入 `TrackLoadedAsset` 写入缓存，没有判断“请求发起时所属生命周期是否仍有效”。
- 当前 `ClearCache` / `ReleaseAsset` 能回收已缓存对象，但对“加载中请求在生命周期失效后晚到完成”的场景没有专门账本与幂等回收机制。

这意味着：如果未来 UI、玩法流程或断线重连流程引入“域级销毁”，当前资源管理器不能区分“有效完成”和“孤儿完成”。

## 3. 问题背景

典型极端时序如下：

1. 某业务界面或业务流程发起异步资源加载。
2. 资源尚未完成加载时，该业务界面被关闭，或者业务流程被强制切换。
3. 业务所属生命周期已经失效，但 Addressables 底层加载仍可能在数百毫秒后完成。
4. 完成结果如果没有被框架识别并拦截，就可能进入错误对象、污染缓存归属，或者形成无法及时释放的悬挂引用。

由于 Addressables 对“取消正在进行中的加载”不提供强语义保证，因此本需求的核心不是强行中断底层加载，而是：

- 在回调落地前识别结果是否仍属于有效生命周期。
- 对已经失效的晚到结果进行安全释放和账本清理。
- 保证孤儿结果不会继续向上层业务穿透。

## 4. 目标与非目标

### 4.1 目标

- 在 xFrame 资源框架中引入“资源域”与“异步请求 token”机制。
- 支持业务方以域为单位发起、追踪和销毁异步加载请求。
- 当域已失效时，晚到的异步加载结果必须被识别为“孤儿结果”。
- 孤儿结果必须在 C# 层立即回收，不得进入业务完成分发。
- 所有回收路径必须幂等，避免重复释放、重复通知或账本错乱。
- 为后续 UI、多界面切换、断线重连等场景提供统一的底层能力。

### 4.2 非目标

- 不要求真正取消 Addressables 的底层 IO 或内部异步操作。
- 不在本期处理 Lua 回调、Lua 引用注册表和跨语言幽灵引用问题。
- 不在本期重构现有全部 `IAssetManager` 使用点，只定义可增量接入的新能力。
- 不在本期引入复杂的资源引用计数共享策略；如需支持共享资源复用，应在后续设计中细化。

## 5. 术语定义

### 5.1 资源域

资源域用于表达一组资源请求所属的业务生命周期边界，例如：

- 某个 UI 界面生命周期
- 某个战斗流程生命周期
- 某个活动模块生命周期

资源域销毁后，属于该域的未完成请求都应被视为无效。

### 5.2 代际

代际用于区分同一逻辑域的不同生命周期轮次。即使逻辑名称相同，只要经历“关闭后重开”，也必须被视为新的一代，旧代回调不得落到新代实例。

### 5.3 加载 Token

加载 Token 是一次异步加载请求的唯一生命周期凭证，至少应携带请求标识、所属域标识、代际和状态信息。

### 5.4 孤儿结果

孤儿结果指底层异步加载已经完成，但其所属资源域或请求 token 已失效，因此不允许继续向上层交付的结果。

## 6. 需求范围

本需求新增的能力应位于 `Assets/xFrame/Runtime/ResourceManager/` 范围内，作为现有 `IAssetManager` 之上的资源域异步扩展层，或作为 `AddressableAssetManager` 的增强能力。

建议优先保持现有 `IAssetManager` 接口兼容，并新增面向域生命周期的异步加载接口，而不是直接破坏现有同步/异步加载接口。

## 7. 功能需求

### 7.1 资源域模型

框架必须提供资源域对象，至少包含以下能力：

- 唯一域标识 `DomainId`
- 当前代际 `Generation`
- 生命周期状态 `IsAlive`
- 域销毁接口

域销毁后：

- `IsAlive` 必须立刻变为 `false`
- 同域未完成请求必须被统一标记为失效
- 同域账本索引必须进入清理流程

### 7.2 异步加载 Token

每次域内异步加载都必须创建唯一 token，至少记录：

- `RequestId`
- `DomainId`
- `Generation`
- `Address`
- `AssetType`
- `Status`
- `IsCanceled` 或等效失效标记
- `IsCompleted`

要求：

- token 只能由资源框架维护，业务层不可任意篡改。
- token 必须能在异步完成时反查所属域和请求状态。
- token 必须支持“请求已失效但底层仍未完成”的中间状态。

### 7.3 请求账本与反查索引

框架必须维护请求账本，至少支持以下索引关系：

- `RequestId -> LoadToken/RequestRecord`
- `DomainId -> RequestId 集合`

如底层需要持有 `AsyncOperationHandle`，还应能反查：

- `RequestId -> HandleRecord`

要求：

- 域销毁时可以快速找到该域全部未完成请求。
- 回调完成时可以快速找到当前请求记录。
- 账本清理必须支持幂等重复调用。

### 7.4 域绑定异步加载接口

框架必须提供带资源域参数的异步加载入口。建议形式包括但不限于：

```csharp
LoadAssetAsync<T>(ResourceDomain domain, string address)
LoadAssetAsync(ResourceDomain domain, string address, Type type)
```

或返回带 token 的请求对象，例如：

```csharp
ResourceLoadRequest<T> LoadAssetAsync<T>(ResourceDomain domain, string address)
```

要求：

- 请求创建时必须绑定域标识和代际。
- 完成分发不得绕过 token 校验。
- 不允许业务层直接把底层 `AsyncOperationHandle` 当作有效性依据。

### 7.5 域销毁拦截

当资源域被销毁时，框架必须：

- 将该域所有未完成请求标记为失效
- 阻止这些请求后续进入成功完成分发
- 对已经完成但尚未清账的请求执行清理
- 移除 `DomainId -> RequestId 集合` 的归属关系

注意：

- 域销毁不要求真正中断 Addressables 底层加载。
- 域销毁后的请求状态必须可区分“已取消未完成”和“完成后被孤儿拦截”。

### 7.6 完成回调闸门

所有异步加载结果在进入业务层前，必须先经过统一的有效性校验。

最少校验项如下：

- token 是否存在
- token 是否已被标记失效
- token 是否已完成或已被消费
- `DomainId` 是否仍存在
- 域当前 `Generation` 是否与 token 创建时一致
- 域是否仍处于 `IsAlive == true`

只要任一条件不满足，该结果必须被判定为孤儿结果。

### 7.7 孤儿结果回收

当异步完成结果被判定为孤儿结果时，框架必须在 C# 层执行以下操作：

- 不向业务完成路径继续分发
- 若底层句柄或结果对象已创建，立即执行安全释放
- 从请求账本和域索引中移除该请求
- 记录孤儿原因和必要上下文

要求：

- 回收逻辑必须幂等
- 不允许因为重复回收导致异常
- 不允许把孤儿结果放入全局缓存，除非后续设计明确支持跨域共享并有独立引用计数模型

### 7.8 缓存写入约束

当前 `AddressableAssetManager` 在加载完成后会直接缓存资源。本需求要求新增约束：

- 只有通过域有效性校验的结果，才允许进入缓存
- 已被判定为孤儿的结果不得写入 `_assetCache`
- 已失效请求不得污染 `_assetToAddressMap`

### 7.9 幂等清理

框架必须支持以下重复路径而不出错：

- 域销毁时清理一次
- 异步完成时再次清理一次
- 外部异常流程导致兜底清理再次执行

幂等清理至少要保证：

- 不重复释放同一个底层句柄
- 不重复删除已移除的账本记录时报错
- 不重复触发业务完成通知

### 7.10 诊断与日志

框架必须为孤儿结果提供诊断信息，至少记录：

- `RequestId`
- `DomainId`
- `Generation`
- `Address`
- 资源类型
- 发起时间
- 完成时间或耗时
- 孤儿原因

建议孤儿原因至少区分：

- `DomainDestroyed`
- `GenerationMismatch`
- `RequestCanceled`
- `RequestRecordMissing`
- `AlreadyCompleted`

日志输出优先接入 xFrame 现有日志体系，不应继续扩散 `Debug.LogError` 风格的裸调用。

## 8. 接口与设计约束

### 8.1 兼容现有接口

应尽量保留现有 [IAssetManager](/mnt/c/UnityProject/xFrame/Assets/xFrame/Runtime/ResourceManager/IAssetManager.cs) 接口，新增能力建议通过以下方式接入：

- 新增 `IResourceDomain`
- 新增 `IResourceDomainAssetManager`
- 或在现有 `IAssetManager` 上增加带域参数的重载

若扩展现有接口，会影响当前注入点和调用方，需评估对 `UIManager` 等现有模块的迁移成本。

### 8.2 Addressables 封装边界

业务层不应直接接触 `AsyncOperationHandle`。相关对象只能存在于资源框架内部，用于：

- 等待完成
- 释放资源
- 账本追踪

### 8.3 线程与主线程约束

由于 Unity 对对象访问存在主线程约束，完成判定、缓存写入和释放动作应明确运行线程语义。若释放必须回主线程，应在设计中定义统一调度策略。

## 9. 验收标准

以下场景必须通过自动化测试验证：

### 9.1 基础生命周期

- 域存活时发起异步加载，资源完成后可正常返回结果。
- 同一域销毁后重新创建新一代域，旧代请求结果不得落到新代。

### 9.2 极端时序

- 异步加载未完成时域被销毁，完成结果到达后应被判定为孤儿并自动释放。
- 域销毁与完成回调几乎同时发生时，清理逻辑不得抛异常。

### 9.3 缓存正确性

- 孤儿结果不得进入缓存。
- 有效结果才允许进入缓存。

### 9.4 幂等性

- 同一请求被重复清理多次时不报错、不重复释放。

### 9.5 并发性

- 同一域内多个并发请求可独立被标记失效。
- 多域并发加载时，一个域销毁不得影响其他域的有效请求。

## 10. 测试需求

建议在 `Assets/xFrame/Tests/EditMode/ResourceManagerTests/` 新增至少以下回归测试：

- `LoadAssetAsync_DomainAlive_ShouldDispatchResult`
- `LoadAssetAsync_DomainDestroyedBeforeComplete_ShouldReleaseOrphan`
- `LoadAssetAsync_GenerationChanged_ShouldDropStaleResult`
- `CleanupOrphan_ShouldBeIdempotent`
- `LoadAssetAsync_MultiDomain_ShouldIsolateLifecycle`

如果需要验证 Unity 对象释放、副作用或主线程行为，再补充对应 PlayMode 测试。

## 11. 分阶段落地建议

### 第一阶段

- 引入 `ResourceDomain`、`LoadToken`、请求账本与统一完成闸门。
- 在不改动现有大多数调用方的前提下，增加新的域绑定异步加载入口。
- 先覆盖 UI 等最容易出现快速开关的场景。

### 第二阶段

- 统一日志与诊断事件输出。
- 补齐缓存策略与共享资源策略边界。
- 评估是否需要将现有无域异步接口逐步收敛到新模型。

## 12. 实现注意事项

- 当前 [AddressableAssetManager](/mnt/c/UnityProject/xFrame/Assets/xFrame/Runtime/ResourceManager/AddressableAssetManager.cs) 是按 `address` 维度合并 inflight 请求，这与“请求级 token”设计存在模型差异，实现时需要显式处理。
- 如果多个域请求同一地址，不能再仅依赖 `_inflightLoads[address] = Task<Object>` 判断生命周期，因为同一个底层加载结果可能对应多个上层请求 token。
- 因此实际实现很可能需要把“底层加载任务”与“上层请求 token”拆成两层结构：
  - 底层共享加载记录
  - 上层域绑定请求记录

只有这样，才能在共享底层加载成本的同时，对不同域做独立的完成判定与孤儿拦截。

## 13. 结论

xFrame 当前资源管理模块已具备基础异步加载、缓存和释放能力，但尚未支持“域级生命周期约束下的异步结果拦截”。在 UI 快速开关、场景切换、断线重连等极端时序下，这会成为框架级缺口。

本需求的核心不是取消 Addressables，而是建立一套可识别、可拦截、可回收、可诊断的请求生命周期控制机制，使晚到结果最多造成一次无效加载成本，但不会污染业务状态和资源账本。
