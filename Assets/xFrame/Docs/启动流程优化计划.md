# 启动流程优化计划

## 目标

- 统一游戏启动入口，避免 `UnityStartupEntry` 与 `xFrameApplication/xFrameBootstrapper` 双轨并存。
- 让当前实际生效的启动链路直接接入 `xFrameLifetimeScope`，保证日志、资源、网络、调度等基础服务可被启动任务使用。
- 逐步将示例型启动骨架补齐为可运行、可回归的正式启动系统。

## 优先级

1. 打通 `UnityStartupEntry -> xFrameLifetimeScope`，让新启动链路拥有统一 DI 容器与基础运行时上下文。
2. 将 `UnityStartupInstallerExample` 中的占位任务替换为真实启动任务，并明确任务依赖、失败策略与输出。
3. 补齐启动完成/失败的统一事件或状态出口，让 Lobby、登录态和首场景加载可以基于单一信号接入。
4. 完善启动 UI，补齐加载态、错误态、重试与退出交互。
5. 为 `MainScene` 启动链路补充 PlayMode 回归测试，覆盖自动启动、失败重试与环境分支。
6. 在新链路稳定后，将旧入口标记为兼容层或逐步下线，避免后续维护分叉。

## 当前进度

- 第一项已完成：新增纯 C# `StartupRuntime`，由 `UnityStartupEntry` 和 `UnityStartupCompositionRoot` 提供最小 Unity 桥接，并统一接入 `LifetimeScope`。
- 第二项进行中：已引入 `StartupTaskContext`、`IStartupServiceResolver` 和多类启动扩展接口，开始用真实的日志初始化、本地环境信息加载和基于端点提供器的网络连接替换纯占位任务。
- 第二项剩余工作：将更新、SDK、认证和流转能力接到业务侧实现，并补齐失败码与场景级验证。
- 第三项进行中：已新增 `IStartupLifecycleState` / `IStartupLifecycleSink` 作为统一启动状态出口，成功、失败、停止都通过同一纯 C# 状态流发布；同时支持 `IStartupLifecycleHandler` 业务处理器、默认业务流转处理器与事件总线桥接。
- 第四项已具备基础承接能力：`UnityStartupView` 已实现统一错误展示接口，当前错误出口能够直接复用启动视图；后续主要剩余正式 UI 表现与交互打磨。
- 第五项已补充基础 PlayMode 回归：新增 `StartupEntryPlayModeTests`，覆盖自动启动成功链路、失败展示链路与 `DevSkipToBattle` 环境分支，并验证 Unity 薄层能够在真实帧循环中驱动纯 C# 启动运行时。
- 为支撑第五项落地，已修复默认 `xFrameLifetimeScope` 在 PlayMode 下的若干 VContainer 重复契约注册问题，并增强 `UnityStartupCompositionRoot` 对容器构建时机的兜底处理。
- 第六项进行中：已新增旧入口兼容工具，对 `xFrameApplication` 与 `xFrameBootstrapper` 做 legacy 标记和共存告警，降低新旧入口误混用的风险。

## 第一阶段实施要点

- 新增纯 C# `StartupRuntime` 作为启动运行时，将启动编排与宿主环境桥接解耦。
- Unity 侧仅保留 `UnityStartupEntry` 与 `UnityStartupCompositionRoot` 两层薄封装。
- 在 Unity 装配根初始化阶段确保场景内存在且仅存在一个 `LifetimeScope`。
- 优先复用场景中的已有 `LifetimeScope`，不存在时再创建默认的 `xFrameLifetimeScope`。
- 新链路创建的 `LifetimeScope` 需要支持 `DontDestroyOnLoad`，与启动根对象保持一致的生命周期。
- `UnityStartupEntry` 对外暴露当前绑定的 `LifetimeScope`，便于后续任务与测试访问。
- 为该行为补充 EditMode 回归测试，覆盖“自动创建”和“复用已有容器”两个基础场景。

## 第二阶段当前落地方向

- 以 `StartupTaskContext` 作为纯 C# 启动上下文，统一承载服务解析与共享状态。
- 启动任务通过接口扩展点获取更新、SDK、认证、流转、网络端点能力，而不是直接依赖 Unity 对象。
- 先将日志初始化、本地环境信息加载升级为真实任务；网络连接改为基于 `IStartupNetworkEndpointProvider` 的环境感知任务。
- 更新、SDK、认证、流转任务统一改为接口扩展点，业务实现接入前由启动失败策略决定是跳过还是中断。

## 第三阶段当前落地方向

- 启动运行时在纯 C# 层发布 `Starting / Succeeded / Failed / Stopped` 四类生命周期快照。
- 默认生命周期状态存储通过 `xFrameLifetimeScope` 注入，业务层可直接解析 `IStartupLifecycleState` 订阅，而不依赖 `MonoBehaviour`。
- 启动运行时会自动分发已注册的 `IStartupLifecycleHandler`，用于承接启动成功后的业务入口或失败后的补偿逻辑。
- 默认 `StartupBusinessFlowLifecycleHandler` 会在启动成功后补齐 Lobby/测试战斗流转，并在失败时委托给 `IStartupFailureHandler`。
- 兼容层通过 `StartupLifecycleEventBusBridge` 将生命周期快照单向桥接到现有 `xFrameEventBus`。

## 第五阶段当前落地方向

- PlayMode 回归优先围绕 `UnityStartupEntry` 这层宿主薄封装展开，而不是将测试强耦合到具体业务场景资源。
- 自动启动验证关注“进入帧循环后是否自动执行启动任务并发布成功状态”。
- 失败路径验证关注“统一失败出口是否通过 `IStartupErrorPresentationService` 回落到 `UnityStartupView`”。
- 若后续需要补环境分支，可继续在 PlayMode 测试内扩展 `DevSkipToBattle` 与自定义 `IStartupFlowService` 桩，而无需回退到大场景耦合测试。

## 第六阶段当前落地方向

- 旧入口暂不做破坏性删除，先作为兼容层保留，避免影响历史接入和现有测试。
- `xFrameApplication` 与 `xFrameBootstrapper` 在场景中检测到 `UnityStartupEntry` 共存时，会输出统一告警，提示新项目应改用 `UnityStartupEntry` 作为唯一启动入口。
- 组件菜单层级已改为 `xFrame/Legacy/*`，降低编辑器误挂载概率。
